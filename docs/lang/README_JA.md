## 👋 私について

こんにちは、CHT.です。香港理工大学を卒業しました。プロダクトマネージャーとして働きながら、興味駆動の開発とデータ分析エンジニアリングにも取り組んでいます。

私は実際の課題から始めることが好きです。繰り返す作業、曖昧なプロダクト上の問い、あるいは文脈が失われ続けるワークフローをプロダクトに整理し、コードとデータで形にして検証できるようにします。

このプロフィールには、私の3つの活動軸をまとめています。どのリポジトリも具体的な課題から始まり、トレードオフ、根拠、そしてプロダクトの将来像に対する私の判断を反映しています。

<details>
<summary><strong>clip-md</strong> — 大切な内容をコピーし、Markdownとして残す。</summary>

<br>

私は、役立つAIコンテンツを保存するときに何度も感じた摩擦から`clip-md`をつくりました。ChatGPT、Claudeなどのツールは残しておきたい回答をよく生み出し、私はそれらをローカルのナレッジベースに加えたいと考えていました。以前の流れでは、エディターを開き、新しいファイルを作成し、保存先とファイル名を決め、最後に保存する必要がありました。その手順は、残したかった思考の流れを中断するほど長いものでした。

そこで中核となる操作を、テキストをコピーし、ポインターの近くに現れるコントロールを使い、そのままローカルMarkdownとして保存するという小さな動作にまとめました。自動命名も意図的な選択です。アプリはキーワードを抽出し、抽出がうまくいかない場合はフォールバックの連鎖を使うため、ファイルの読みやすさと検索性を保ちながら、判断を1つ減らせます。

私は`clip-md`を、AIを頻繁に使う人やナレッジワーカーに役立つものにしたいと考えています。軽量さを保ちながら、信頼性、各プラットフォームとの適合、署名付き配布、そしてデスクトップツールを長く使い続けるための細かな体験を改善していきます。

</details>

<details>
<summary><strong>gmail-inbox-system-public</strong> — 1つの受信箱に複数の意味を保ち、自動化を制御する。</summary>

<br>

私は、複数のメールボックスをGmailに集約した経験からGmail Inbox Systemをつくりました。切り替えは減ったものの、受信箱から送信元や対応状況の意味が失われ始めました。アプリ通知、請求書、セキュリティ警告、個人メール、保留中の作業が1つの流れに入り、以前のラベルは送信元、内容の種類、状態を混在させていたため、時間とともに保守が難しくなっていました。

そこでラベル体系を、アプリケーション、種類、送信元、状態という組み合わせ可能な軸に分け、それぞれが1つの意味を表すようにしました。各メールは、どこから届き、何を含み、どの対応が必要かを同時に保持できます。分類、プレビュー、保持期間、保護ルールをアーカイブ前の明確なゲートにし、判断がつかないメールは見える場所に残します。

私はこの仕組みを、Gmailを統一された入口とし、将来はAIによるデイリーブリーフィングも加えた個人メールハブに育てたいと考えています。公開リポジトリには再利用可能でプライバシーに配慮したルールと例だけを収録し、個人メール、アカウント認証情報、実際のメールボックスへのアクセスは含めていません。

</details>

<details>
<summary><strong>okx-btc-daily-report-public</strong> — 自分の積立戦略を、自分の基準で測る。</summary>

<br>

私は自分のBTC積立戦略を実践しているため、OKX BTC Daily Reportをつくりました。取引所の組み込み積立プランは、私が求めるルールやペースに合わず、固定されたダッシュボードでは、30日間のパフォーマンス、計画額と実際の投資額、約定と手数料を反映した結果など、私が知りたい問いに答えられませんでした。

このワークフローは読み取り専用アクセスで取引データを取得し、私自身の定義に沿って日次、7日、30日、累計の表示を再計算します。分割約定、手数料、建値通貨の違いによって手作業の記録は時間とともにずれるため、同じルールを継続的に実行する必要があります。既存の習慣の中でレポートを受け取り、別のダッシュボードを管理せずに済むよう、配信方法にはメールを選びました。

私はこれを、より多くの資産、期間、指標、可視化に対応する個人投資データコンソールに育てたいと考えています。公開リポジトリは機密情報を除いた参考実装であり、APIキーを含まず、私のOKXアカウントを閲覧できず、取引も実行しません。

</details>

<details>
<summary><strong>quant-research</strong> — 失敗も含め、根拠を残す。</summary>

<br>

私は投資アイデアを、データによって検証できる仮説へ変えるために`quant-research`をつくりました。戦略はもっともらしく見えても、都合のよい1つの期間だけに依存していることがあります。私は、どのデータを使い、どうシグナルを形成し、リターンがどこから生まれ、期間、パラメーター、モデルを変えたあとにどれだけ効果が残るのかを確かめたいと考えています。

特に重視しているのは、過学習と分断された研究過程です。パラメーター探索は魅力的なサンプル内結果を生む一方、コード、データ、一時的な出力のつながりは次第に失われます。私は弱いファクター、失敗したモデル、性能が低下した戦略も残し、リターンと並べてコスト、要因分析、ドローダウン、サンプル外の挙動を評価します。失敗した実験からも、どの道筋に持続的な根拠が欠けているかを学べます。

私は`quant-research`を、スナップショット、仮説、シグナル、バックテスト、検証、記録された知見を扱う再利用可能なフレームワークに育てたいと考えています。新しい市場やモデルを同じ構造に加えられるようにし、個人研究を蓄積可能で、比較でき、振り返れるものにしていきます。

</details>

<details>
<summary><strong>crypto-trading-research</strong> — アクティブなBTC取引は、複雑さに見合う成果を生めるか。</summary>

<br>

私は自分の意思決定に直接関わる問いを軸に`crypto-trading-research`をつくりました。手数料、スリッページ、売買回転、注意力まで数えたとき、BTCの長期保有はすでに最も低コストで手間の少ない戦略なのか。アクティブ取引や機械学習は高度に見えますが、複雑さだけで追加価値は証明できません。

私は長期保有を常設の基準に置き、古典的な指標、統計シグナル、機械学習モデルを順に比較します。頻繁な売買で見かけ上の優位性が消える可能性があるため、コストは明示します。期間と手数料の仮定も見える状態にし、モデル指標を実際の戦略リターンに変換して確かめます。MA、RSI、Bollinger Bands、XGBoostの実験は、結果が弱い場合もリポジトリに残します。

私はこのプロジェクトを、新しいデータや相場局面でも同じ問いを検証し続けるBTC意思決定の根拠集に育てたいと考えています。すべての結論をサンプル期間、コストの仮定、検証方法に結びつけ、ほかの学習者が研究過程全体を振り返れるようにします。

</details>

<details>
<summary><strong>ecommerce-user-analysis</strong> — 集計指標の中に隠れた顧客を見る。</summary>

<br>

私は取引記録が意味のある顧客セグメントへどう変わるのかを理解するために`ecommerce-user-analysis`をつくりました。売上、注文数、平均注文額は事業全体を表しますが、ロイヤル顧客、離れつつある顧客、有望な新規顧客、異なる戦略が必要な顧客の違いを集計の中に圧縮してしまいます。

私はRFMで解釈しやすい事業セグメントをつくり、続いてK-Meansでデータの自然な構造を調べます。この2つを組み合わせることで、明確な実務言語を保ちながら、もう1つの分析視点を加えられます。プロジェクトは100万件を超える過去の取引を、クリーニング、探索、セグメント化、可視化へ進め、最後に維持、再活性化、転換の仮説へ変換します。

私はこれを、セグメントの変化、各グループに重要な指標、行動を実際に変える成長実験を追跡するEコマース成長分析コンソールに育てたいと考えています。

</details>

<details>
<summary><strong>ai-chat-analytics</strong> — 会話のシグナルをプロダクトの優先順位に変える。</summary>

<br>

私は`ai-chat-analytics`をつくりました。いいね、低評価、再試行、追加質問だけでは、AIプロダクトチームが何を最初に直すべきか、ユーザーがなぜ不満なのか、改善によって体験が変わったかを直接判断できないためです。「全般的な品質」のような広いラベルは問題の場所を示せても、原因を明確にできないことがあります。

作業を合成データ生成、行動分析、問題の帰属、追加質問のシグナル、知見の要約に分けました。固定シードを使うことで、実際の会話を公開せずに公開スナップショットを再現できます。透明なキーワードルールで説明可能な基準をつくり、その限界も明示しています。現在の低評価事例の71.2%はまだ包括的なラベルに入り、手作業による検証も完了していません。

私はこのプロジェクトをAIプロダクト責任者に役立て、継続的な指標、帰属の品質、アラート、実験結果を1か所につなぐAI体験モニタリングコンソールに育てたいと考えています。

</details>

<details>
<summary><strong>readme-craft</strong> — 公開ドキュメントで、リポジトリができることを証明する。</summary>

<br>

私は公開プロジェクトで同じ公開判断を何度も繰り返した経験から`readme-craft`をつくりました。ヒーロー領域の構成、バッジの色、セクション順、多言語ナビゲーション、表の幅、Mermaidのサイズ、プライバシーチェックには、その都度手作業の判断が必要でした。複数のリポジトリで品質を安定して保てる、再利用可能な公開基準が欲しいと考えました。

中核となる原則は、根拠に基づく設計です。このSkillは文章を書く前に、コード、マニフェスト、リリース、ワークフロー、ライセンス、リポジトリの状態を確認します。バリデーターはリンク、アンカー、言語間の等価性、図、個人情報を含むパスを確認し、レンダリングしたプレビューでソースだけでは判断できないレイアウトを検証します。

私は`readme-craft`を独立開発者やメンテナーに役立て、将来はリポジトリの変更やコンテンツのずれを継続的に検出するREADME保守システムに育てたいと考えています。

</details>

<details>
<summary><strong>desktop-organizer</strong> — 何かを動かす前に、すべての移動を確認する。</summary>

<br>

私は、文書、スクリーンショット、インストーラー、アーカイブ、コードプロジェクトでいっぱいになったWindows Desktopを整理したいという自分の必要から`desktop-organizer`をつくりました。手作業の整理には判断の繰り返しが必要で、直接的な自動化には誤分類、上書き、意図しないファイル操作のリスクがありました。

ワークフローは、検査、計画、承認、実行、検証の順です。既定モードではdry-runだけを表示し、ファイルの移動には明示的な承認が必要です。削除も上書きも行わず、競合が見つかれば実行前に停止します。ショートカットとシステムファイルは元の場所に残し、判断が難しい項目は確認用の受信箱に入れ、すべての処理をローカルで完結させます。

私はこれをWindows Desktopを頻繁に使う人に役立て、それぞれの分類体系、ルール、例外リストを保持できる個別化された分類システムに育てたいと考えています。

</details>

<details>
<summary><strong>become-master</strong> — 5分で、会話に入る足場を見つける。</summary>

<br>

私は、テクノロジー分野で働く親しい友人が会話に入るのに苦労している姿を見て`become-master`をつくりました。彼は能力も好奇心もありましたが、同僚や好意を持つ相手との会話では、共通の話題をすぐに見つけられず、話が止まることがありました。検索からは大量の情報を得られても、優先順位はほとんど得られませんでした。

このSkillは分野と実際の状況から始まり、用語、プロダクト、判断、会話の糸口、よくある落とし穴、立て直し方を整理します。最も重要な選択は強制的な圧縮です。広い分野を、会話の直前に見直せる18ノードの記憶カードにまとめます。私が重視する方向性は、誠実な準備、よりよく聞くこと、より自然な質問です。

私はこれを、関係性と状況を使って共通の関心を見つけ、会話の始め方を準備し、自然に会話を続けられるよう支援する個人向け会話準備アシスタントに育てたいと考えています。

</details>

<details>
<summary><strong>ai-dev-context-framework</strong> — セッションをまたいで、コーディングAgentの方向をそろえる。</summary>

<br>

私は、コーディングAgentがセッションをまたぐと連続性を失うことがあるため、`ai-dev-context-framework`をつくりました。プロダクト意図、技術的制約、アーキテクチャ上の選択、タスク状態がチャットや一時ファイルに散らばります。プロジェクトが大きくなると、日常的な進捗更新が、ユーザーの管理下に残すべき意思決定にまで及ぶことがあります。

私はコンテキスト層にMarkdown契約を選びました。プロダクトの議論を`SPEC.md`にし、そこからAgentの入口、タスクキュー、さらに複雑さに応じてアーキテクチャ、データ、デザイン、課題、意思決定、引き継ぎのファイルを作成します。ロック領域と生成領域で保守境界を表し、Full版とLite版で異なる規模のプロジェクトに対応します。現在のガードレールは、Agentが従うことを前提とした規約です。

私はこのフレームワークを複雑なプロジェクトのメンテナーに役立て、コンテキストの一貫性、権限境界、意思決定記録、信頼できる再開地点を確認するAgentプロジェクトガバナンス層に育てたいと考えています。

</details>

<a id="principles"></a>
## 🧭 私のつくり方

| 原則 | 実践での意味 |
|---|---|
| **自分が感じた摩擦から始める** | プロジェクトは、私が具体的に説明できる課題から始める |
| **根拠を見える状態にする** | 仮説、コスト、失敗した実験、限界を結果と並べる |
| **制御の境界を明確にする** | ローカルファイル、個人アカウント、取り消せない判断には明確な境界を設ける |
| **インターフェースを圧縮する** | 最小限で役立つ操作が、最も長く使える形になることが多い |
| **再利用を前提に設計する** | 繰り返すワークフローをツール、フレームワーク、Skillに変える |
| **長期で考える** | 時間とともに積み重なるシステム、知識、資産を大切にする |
