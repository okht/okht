<details>
<summary><strong>clip-md</strong> — 중요한 내용을 복사하고 Markdown으로 남깁니다.</summary>

<br>

저는 유용한 AI 콘텐츠를 저장할 때 반복해서 느낀 마찰에서`clip-md`를 만들었습니다. ChatGPT, Claude 같은 도구는 보관할 가치가 있는 답변을 자주 만들고, 저는 그 답변을 로컬 지식 기반의 일부로 쌓고 싶었습니다. 기존 흐름에서는 편집기를 열고, 새 파일을 만들고, 저장 위치와 파일 이름을 정한 뒤 저장해야 했습니다. 이 과정은 보존하려던 생각의 흐름을 끊을 만큼 길었습니다.

핵심 상호작용은 작은 동작 하나로 줄였습니다. 텍스트를 복사하고, 포인터 근처에 나타나는 컨트롤을 사용해 로컬 Markdown으로 바로 저장합니다. 자동 이름 지정도 의도적으로 선택했습니다. 앱은 키워드를 추출하고 추출 결과가 약할 때 폴백 체인을 사용하므로, 파일의 가독성과 검색 가능성을 유지하면서 판단 하나를 덜 수 있습니다.

저는`clip-md`가 AI를 자주 사용하는 사람과 지식 노동자에게 도움이 되기를 바랍니다. 가벼운 방향을 유지하면서 신뢰성, 플랫폼 적합성, 서명된 배포, 데스크톱 도구를 오랫동안 유용하게 만드는 세심한 경험을 개선해 나가고 싶습니다.

</details>

<details>
<summary><strong>gmail-inbox-system-public</strong> — 하나의 받은편지함에 여러 의미를 보존하고 자동화를 통제합니다.</summary>

<br>

저는 여러 메일함을 Gmail로 모은 뒤 Gmail Inbox System을 만들었습니다. 전환 횟수는 줄었지만, 받은편지함에서 출처와 조치 상태의 의미가 흐려지기 시작했습니다. 앱 알림, 청구서, 보안 경고, 개인 메시지, 대기 중인 업무가 한 흐름으로 들어왔고, 기존 라벨은 출처, 콘텐츠 유형, 상태를 섞어 표현해 시간이 갈수록 관리하기 어려워졌습니다.

라벨 체계를 애플리케이션, 유형, 출처, 상태라는 조합 가능한 차원으로 분리해 각 차원이 하나의 의미를 나타내도록 했습니다. 한 메일에 어디서 왔는지, 무엇을 담고 있는지, 어떤 조치가 필요한지를 함께 보존할 수 있습니다. 분류, 미리보기, 보존 기간, 보호 규칙은 보관 처리 전의 명확한 관문을 만들고, 판단하기 어려운 메일은 계속 보이는 곳에 둡니다.

저는 이 시스템을 Gmail을 통합 입구로 사용하고, 향후 AI 일일 브리핑을 더하는 개인 이메일 허브로 키우고 싶습니다. 공개 저장소에는 재사용할 수 있고 개인정보를 배려한 규칙과 예시만 담았으며, 개인 메일, 계정 자격 증명, 실제 메일함 접근 권한은 포함하지 않습니다.

</details>

<details>
<summary><strong>okx-btc-daily-report-public</strong> — 제 적립식 투자 전략을 제 기준으로 측정합니다.</summary>

<br>

저는 제 BTC 적립식 투자 전략을 따르기 때문에 OKX BTC Daily Report를 만들었습니다. 거래소의 기본 적립식 투자 계획은 제가 원하는 규칙과 주기에 맞지 않았고, 고정된 대시보드는 30일 성과, 계획 투자액과 실제 투자액의 차이, 체결과 수수료를 반영한 결과처럼 제가 중요하게 보는 질문에 답하지 못했습니다.

이 워크플로는 읽기 전용 접근으로 거래 데이터를 가져와 제가 정한 정의에 따라 일간, 7일, 30일, 누적 화면을 다시 계산합니다. 분할 체결, 수수료, 호가 통화 때문에 수동 기록은 시간이 지날수록 어긋날 수 있어 동일한 규칙을 꾸준히 실행해야 합니다. 별도 대시보드를 관리할 필요 없이 기존 일상 안에서 보고서를 받도록 이메일 전달 방식을 선택했습니다.

저는 이를 더 많은 자산, 기간, 지표, 시각화를 다루는 개인 투자 데이터 콘솔로 발전시키고 싶습니다. 공개 저장소는 민감한 정보를 제거한 참고 구현입니다. API 키가 없고, 제 OKX 계정을 볼 수 없으며, 거래를 실행하지 않습니다.

</details>

> 투자 및 트레이딩 프로젝트는 개인 연구 기록입니다. 투자 조언을 제공하지 않습니다.

<details>
<summary><strong>quant-research</strong> — 실패까지 포함해 근거를 남깁니다.</summary>

<br>

저는 투자 아이디어를 데이터가 검증할 수 있는 가설로 바꾸기 위해`quant-research`를 만들었습니다. 전략이 합리적으로 보여도 한 번의 유리한 기간에 의존할 수 있습니다. 어떤 데이터를 사용했고, 어떻게 신호를 만들었고, 수익이 어디서 나왔으며, 기간, 매개변수, 모델이 달라진 뒤에도 얼마나 남는지 확인하고 싶었습니다.

저는 과적합과 분절된 연구 과정에 특히 주의를 기울입니다. 매개변수 탐색은 매력적인 표본 내 결과를 만들 수 있지만, 코드, 데이터, 임시 출력 사이의 연결은 서서히 사라질 수 있습니다. 약한 팩터, 실패한 모델, 성능이 저하된 전략도 보존하고, 수익과 함께 비용, 귀속, 낙폭, 표본 외 행동을 평가합니다. 실패한 실험도 어떤 경로에 지속 가능한 근거가 부족한지 알려줍니다.

저는`quant-research`를 스냅샷, 가설, 신호, 백테스트, 검증, 기록된 결과를 위한 재사용 가능한 프레임워크로 키우고 싶습니다. 새로운 시장과 모델을 같은 구조에 넣어 개인 연구를 축적하고, 비교하고, 다시 검토할 수 있게 만들고자 합니다.

</details>

<details>
<summary><strong>crypto-trading-research</strong> — 적극적인 BTC 거래는 복잡성에 걸맞은 성과를 낼 수 있을까요?</summary>

<br>

저는 제 의사결정에 직접 영향을 주는 질문을 중심으로`crypto-trading-research`를 만들었습니다. 수수료, 슬리피지, 회전율, 주의력까지 계산했을 때 장기 BTC 보유가 이미 비용과 노력이 가장 적은 전략일까요? 적극적인 거래와 머신러닝은 정교해 보일 수 있지만, 복잡성만으로 추가 가치를 증명할 수는 없습니다.

장기 보유를 지속적인 기준선으로 두고 고전 지표, 통계 신호, 머신러닝 모델을 차례로 비교합니다. 잦은 매매가 장부상의 우위를 지울 수 있으므로 비용을 명시합니다. 기간과 수수료 가정도 보이는 상태로 유지하고, 모델 지표를 실제 전략 수익으로 바꿔 확인합니다. MA, RSI, Bollinger Bands, XGBoost 실험은 결과가 약해도 저장소에 남깁니다.

저는 이 프로젝트를 새로운 데이터와 시장 국면에서도 같은 질문을 계속 검증하는 BTC 의사결정 근거 모음으로 키우고 싶습니다. 모든 결론을 표본 기간, 비용 가정, 검증 방법에 연결해 다른 학습자도 전체 연구 과정을 살펴볼 수 있게 합니다.

</details>

<details>
<summary><strong>ecommerce-user-analysis</strong> — 집계 지표 안에 숨은 고객을 봅니다.</summary>

<br>

저는 거래 기록이 어떻게 의미 있는 고객 세그먼트가 되는지 이해하기 위해`ecommerce-user-analysis`를 만들었습니다. 매출, 주문 수, 평균 주문 금액은 전체 사업을 설명하지만, 충성 고객, 이탈 중인 고객, 유망한 신규 고객, 다른 전략이 필요한 사람 사이의 차이를 하나의 집계값 안에 압축합니다.

RFM으로 해석 가능한 사업 세그먼트를 만들고, 이어서 K-Means로 데이터의 자연스러운 구조를 살펴봅니다. 두 방법을 함께 사용하면 명확한 실무 언어를 유지하면서 두 번째 분석 관점을 더할 수 있습니다. 이 프로젝트는 100만 건이 넘는 과거 거래를 정제, 탐색, 세분화, 시각화로 이어가고, 그 결과를 유지, 재활성화, 전환 가설로 바꿉니다.

저는 이를 세그먼트가 어떻게 변하는지, 각 그룹에 어떤 지표가 중요한지, 어떤 성장 실험이 실제 행동을 바꾸는지 추적하는 전자상거래 성장 분석 콘솔로 발전시키고 싶습니다.

</details>

<details>
<summary><strong>ai-chat-analytics</strong> — 대화 신호를 제품 우선순위로 바꿉니다.</summary>

<br>

저는 좋아요, 싫어요, 재시도, 후속 질문만으로는 AI 제품 팀이 무엇을 먼저 고쳐야 하는지, 사용자가 왜 불만족하는지, 개선이 경험을 바꾸었는지 직접 알기 어렵기 때문에`ai-chat-analytics`를 만들었습니다.「전반적 품질」처럼 넓은 라벨은 문제의 위치를 보여줄 수 있어도 원인은 불분명하게 남을 수 있습니다.

작업을 합성 데이터 생성, 행동 분석, 문제 귀속, 후속 질문 신호, 인사이트 요약으로 나눴습니다. 고정 시드를 사용해 실제 대화를 노출하지 않고도 공개 스냅샷을 재현할 수 있습니다. 투명한 키워드 규칙으로 설명 가능한 기준선을 만들고 그 한계도 드러냅니다. 현재 실패 사례의 71.2%는 여전히 포괄 라벨에 들어가며, 수동 검증도 완료되지 않았습니다.

저는 이 프로젝트가 AI 제품 책임자에게 도움이 되고, 지속적인 지표, 귀속 품질, 알림, 실험 결과를 한곳에 연결하는 AI 경험 모니터링 콘솔로 성장하기를 바랍니다.

</details>

<details>
<summary><strong>readme-craft</strong> — 공개 문서로 저장소가 할 수 있는 일을 증명합니다.</summary>

<br>

저는 공개 프로젝트마다 같은 게시 결정을 반복한 경험에서`readme-craft`를 만들었습니다. 첫 화면 구성, 배지 색상, 섹션 순서, 언어 탐색, 표 너비, Mermaid 크기, 개인정보 점검 모두 매번 수동 판단이 필요했습니다. 여러 저장소에서 품질을 안정적으로 유지할 수 있는 재사용 가능한 게시 기준을 원했습니다.

핵심 원칙은 근거 기반 설계입니다. 이 Skill은 글을 쓰기 전에 코드, 매니페스트, 릴리스, 워크플로, 라이선스, 저장소 상태를 살펴봅니다. 검증기는 링크, 앵커, 언어 간 동등성, 다이어그램, 개인 경로를 확인하고, 렌더링된 미리보기는 소스만으로 판단할 수 없는 레이아웃을 검증합니다.

저는`readme-craft`가 독립 개발자와 유지관리자에게 도움이 되고, 앞으로 저장소 변경과 콘텐츠 불일치를 감지하는 README 유지관리 시스템으로 성장하기를 바랍니다.

</details>

<details>
<summary><strong>desktop-organizer</strong> — 무엇이든 옮기기 전에 모든 이동을 확인합니다.</summary>

<br>

저는 문서, 스크린샷, 설치 프로그램, 압축 파일, 코드 프로젝트로 가득 찬 Windows Desktop을 정리해야 했던 경험에서`desktop-organizer`를 만들었습니다. 수동 정리는 반복적인 판단이 필요했고, 직접 자동화하면 오분류, 덮어쓰기, 의도하지 않은 파일 작업 위험이 생겼습니다.

워크플로는 검사, 계획, 승인, 실행, 검증 순서로 진행됩니다. 기본 모드는 dry-run만 출력하며, 파일 이동에는 명시적인 승인이 필요합니다. 삭제하거나 덮어쓰지 않고, 충돌이 발견되면 일찍 실행을 중단합니다. 바로가기와 시스템 파일은 제자리에 두고, 확실하지 않은 항목은 검토함으로 보내며, 모든 작업은 로컬에서 처리합니다.

저는 이 도구가 Windows Desktop을 많이 사용하는 사람에게 도움이 되고, 각자가 자신의 분류 체계, 규칙, 예외 목록을 유지할 수 있는 개인화된 분류 시스템으로 발전하기를 바랍니다.

</details>

<details>
<summary><strong>become-master</strong> — 5분 안에 대화에 들어갈 발판을 찾습니다.</summary>

<br>

저는 기술 분야에서 일하는 가까운 친구가 대화에 끼어드는 데 어려움을 겪는 모습을 보고`become-master`를 만들었습니다. 그는 유능하고 호기심도 많았지만, 동료나 좋아하는 사람과 이야기할 때 공통 주제를 빨리 찾지 못해 대화가 멈추곤 했습니다. 검색은 많은 정보를 주었지만 우선순위는 거의 주지 못했습니다.

이 Skill은 분야와 실제 상황에서 시작해 용어, 제품, 판단, 대화 소재, 흔한 함정, 회복 방식을 정리합니다. 가장 중요한 선택은 강제 압축입니다. 넓은 분야를 대화 직전에 다시 볼 수 있는 18개 노드의 기억 카드로 마무리합니다. 제가 중요하게 보는 방향은 솔직한 준비, 더 나은 경청, 더 자연스러운 질문입니다.

저는 이를 관계와 상황을 활용해 공통 관심사를 찾고, 대화 시작을 준비하고, 자연스럽게 대화를 이어가도록 돕는 개인 대화 준비 도우미로 발전시키고 싶습니다.

</details>

<details>
<summary><strong>ai-dev-context-framework</strong> — 세션이 바뀌어도 코딩 Agent가 같은 방향을 유지하게 합니다.</summary>

<br>

저는 코딩 Agent가 세션 사이에서 연속성을 잃을 수 있기 때문에`ai-dev-context-framework`를 만들었습니다. 제품 의도, 기술 제약, 아키텍처 선택, 작업 상태가 채팅과 임시 파일에 흩어집니다. 프로젝트가 커지면 일상적인 진행 상황 업데이트가 사용자의 통제 아래 남아야 할 의사결정까지 건드릴 수 있습니다.

저는 컨텍스트 계층에 Markdown 계약을 선택했습니다. 제품 논의를`SPEC.md`로 만들고, 이어서 프레임워크가 Agent 진입점, 작업 대기열, 복잡도에 따라 추가 아키텍처, 데이터, 디자인, 이슈, 의사결정, 인수인계 파일을 만듭니다. 잠금 영역과 생성 영역으로 유지관리 경계를 나타내며, Full과 Lite 버전은 서로 다른 프로젝트 규모를 지원합니다. 현재 가드레일은 Agent의 준수를 전제로 하는 규칙입니다.

저는 이 프레임워크가 복잡한 프로젝트의 유지관리자에게 도움이 되고, 컨텍스트 일관성, 권한 경계, 의사결정 기록, 신뢰할 수 있는 재시작 지점을 확인하는 Agent 프로젝트 거버넌스 계층으로 성장하기를 바랍니다.

</details>

## 🧭 제가 만드는 방식

| 원칙 | 실제 의미 |
|---|---|
| **직접 겪은 마찰에서 시작하기** | 프로젝트는 제가 구체적으로 설명할 수 있는 문제에서 시작해야 합니다 |
| **근거를 보이게 만들기** | 가정, 비용, 실패한 실험, 한계를 결과와 함께 둡니다 |
| **통제 경계를 명확히 하기** | 로컬 파일, 개인 계정, 되돌릴 수 없는 결정에는 명시적 경계가 필요합니다 |
| **인터페이스 압축하기** | 작고 유용한 상호작용이 오래 유지되는 경우가 많습니다 |
| **재사용을 위해 설계하기** | 반복되는 워크플로를 도구, 프레임워크, Skill로 만듭니다 |
| **장기적으로 생각하기** | 시간이 지나며 축적되는 시스템, 지식, 자산을 중요하게 생각합니다 |
